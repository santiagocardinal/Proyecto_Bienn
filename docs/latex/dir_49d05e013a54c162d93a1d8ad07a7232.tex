{\bfseries{Desafíos Técnicos}}


\begin{DoxyItemize}
\item Definir las relaciones entre {\ttfamily Customer}, {\ttfamily Interaction}, {\ttfamily Meeting}, {\ttfamily Sale} y demás clases requirió aplicar correctamente guías de diseño como Expert. La asignación de responsabilidades fue iterativa, con varios ajustes hasta lograr una estructura coherente y mantenible.
\item Mantener las buenas prácticas de programación mientras buscábamos soluciones fue un desafío constante. La presión de resolver problemas nos generaba el temor de incumplir algún principio {\ttfamily SOLID}, patrón de diseño o guía establecida. En momentos de bloqueo, había que frenar y asegurarnos de no sacrificar calidad por velocidad.
\item Implementamos polimorfismo para manejar diferentes tipos de interacciones, lo que permitió extender funcionalidades sin modificar código existente y facilitó la escalabilidad del sistema.
\item La lógica para identificar clientes sin actividad reciente implicó trabajar con {\ttfamily Date\+Time} y {\ttfamily Time\+Span}. Métodos como {\ttfamily Get\+Inactive\+Customers} y {\ttfamily Get\+Recent\+Interaction} requirieron varias iteraciones hasta quedar robustos y reutilizables.
\item Cobertura de pruebas unitarias con {\ttfamily NUnit}, este proceso tomó tiempo considerable pero resultó fundamental para validar comportamientos y detectar errores tempranamente, estableciendo una base sólida para el desarrollo futuro.
\item El uso de {\ttfamily Git\+Hub} como herramienta de gestión nos obligó a aprender manejo de ramas, resolución de conflictos y coordinación de tareas simultáneas. La curva de aprendizaje fue pronunciada pero necesaria.
\end{DoxyItemize}

{\bfseries{Avances Técnicos}}


\begin{DoxyItemize}
\item En esta entrega incorporamos patrones de diseño que aportaron significativamente a la organización y funcionalidad del sistema.
\item El patrón {\ttfamily Facade} se implementó para facilitar la interacción del usuario con el sistema, permitiendo que este no tenga que conocer la estructura interna del programa. A través de la fachada, los usuarios pueden ingresar los datos directamente en formato {\ttfamily string}, simplificando el uso y haciendo más intuitiva la comunicación con las distintas funcionalidades del sistema.
\item Por otro lado, el patrón {\ttfamily Singleton} nos permitió mantener una única instancia de cada manager, garantizando que las listas se manejen de forma estática y global. De esta manera, al acceder a ellas desde cualquier parte del programa, siempre se trabaja con los mismos datos actualizados, evitando duplicaciones y asegurando la coherencia en la gestión de la información.
\item Además, integramos {\ttfamily Doxygen} como herramienta de documentación, lo que nos permitió mantener una descripción clara y profesional del código, detallando clases, métodos y relaciones del sistema. Esto facilita tanto la comprensión interna entre los miembros del equipo como la futura revisión por parte de terceros.
\end{DoxyItemize}
